## Planned Highlights

### Consortium (quality and policy)

**In production code today (HTTP + Redis + unit tests):** weighted consortium scoring with per-tenant trust; `**POST /v1/consortium/feedback`** outcomes (`**false_positive` / `confirmed_fraud`**) feed FP-rate and quality-style fields used when applying `**consortium_score_delta`**; `**POST /v1/consortium/share**`, `**GET /v1/consortium/check/...**`, `**POST /v1/consortium/trust**`; gating via `**CONSORTIUM_***` env and `**consortium_min_tenants**` (see `services/decision-api/src/decision_api/consortium_api.py`, `redis_store.py`, `consortium.py`, `tests/test_consortium.py`).

**Product surface:** there is **no** consortium-specific **Case / React** dashboard in v1.1; “tenant participation” is **configuration + API**, not a separate analyst policy console. **Aggregate decision throughput** is illustrated by `**services/decision-api/static/dashboard.html`** (general events, not consortium KPIs). A fuller **Trust Center / consortium posture UI** is **roadmapped for later trains** (see `[v1.2.0-2026-05-30.md](v1.2.0-2026-05-30.md)`, `[v1.3.0-2026-06-29.md](v1.3.0-2026-06-29.md)`).

### Inference and decision contract (Epics A, B, E—core)

- Versioned `**inference_context` (schema v2)** with `**confidence_tier`** (low / medium / high).
- `**driver_reasons`**: analyst-oriented top drivers (tamper, replay, network, geo, ML, rules, colocation, travel proxy).
- **Replay / tamper context** reflected in derived risks and drivers (ingress replay signals, rule-based replay hints).
- **Co-location and impossible-travel proxies** (`colocation_risk`, `impossible_travel_risk`) from device and velocity/geo heuristics.
- **Velocity in inference**: `velocity_events_5m`, `velocity_events_1h`, `velocity_events_24h` (aggregates include **5m** window).
- `**recommended_action`** on evaluate and audit responses (e.g. `block`, `manual_review`, `step_up_mfa`, `step_up_attestation`) for client step-up hints.

### Location coherence (trunk; strengthens geo story)

- **SDKs:** TypeScript `**enableGeo`** (browser GPS into `device_context.signals`); Python `**ServerSignalCollector`** optional `**enable_ip_geo`** / env `**ENABLE_IP_GEO_LOOKUP**` (public lookup **off** by default).
- **Decision API:** `**merge_session_geo_from_device_and_features`** (`location_context.py`) — session last-known geo, `**sdk:geo_ip_mismatch`** / `**sdk:geo_tz_mismatch`** tags; `**geo_markers**` in `**inference_build**`; `**/v1/ops/calibration-status**` and `**calibration_status**` on `**/v1/ops/governance**`.
- **Graph:** default schema `**Place`** + `**SEEN_AT`** (`graph_service/custom_schema.py`) for quantized place / co-presence–style edges when graph upserts run.
- **Contracts:** `DeviceContext.signals` documents `**geo_*`** and optional `**ip_geo_*`** in `**contracts/openapi/decision-api.yaml`**.

## Release candidate finalization & May handoff policy

**Goal for `v1.1.0`:** ship with **no failing automated release gates** and **no known P0 blockers** on the **demo path** (lite compose + five-minute doc) and the **full stack** profile used in CI (`docker-build` / `stack-smoke` matrix). *“Zero bugs”* is not mathematically provable; **operational definition** = all required CI jobs pass, security workflows complete without blocking failures, and open **P0** issues affecting demo or tagged stack are **closed or explicitly waived** in the checklist above.

**Minimum bar before tagging `v1.1.0`:**

1. `**[ci.yml](../../../.github/workflows/ci.yml)`** — full matrix green on the release candidate commit: **lint (Ruff)**; **frontend** `npm ci` + `npm run build`; `**investigation-agent`** tests; decision-api (coverage gate), case-api, graph-service, Neo4j live job, integration-ingress, SDK Python, GraphQL, event-ingest, analytics-sink, feature-service, ml-scoring; **TypeScript SDK** build; **docker-build**; **stack-smoke** as configured.
2. `**[security-scan.yml](../../../.github/workflows/security-scan.yml)`** and `**[secret-scan.yml](../../../.github/workflows/secret-scan.yml)`** — completed runs on the same commit (Trivy + secret scanning findings gate), or for **early-development beta only** an explicit, time-bound waiver documented in this checklist with owner and closure date.
3. **Manual smoke:** `[sandbox-five-minute.md](../guides/sandbox-five-minute.md)` (evaluate + OSINT + **Case detail explainability** + copilot/mock path as applicable).

**After `v1.1.0` is tagged (finalized build):** routine product and engineering changes that **miss** the 4/30 tag **do not** retroactively alter `v1.1.0`. They ship on the **May 2026 Friday trains** documented in `[release-calendar-2026-05.md](../guides/release-calendar-2026-05.md)` and roll into `**v1.2.0` (`2026-05-30`)** per `[RELEASE_SCHEDULE.md](../../../RELEASE_SCHEDULE.md)`, unless explicitly cherry-picked as a **patch** (document any exception in release notes).

**Local subset (frontend + copilot + TS SDK + Ruff on `services/investigation-agent`):** from repo root, `bash scripts/ci/verify_release_candidate.sh` — does **not** replace `**ruff check .`** on the whole repo, decision-api coverage, Docker build, stack-smoke, or other service pytests (those stay on **GitHub Actions `ci.yml`**).

## Maintainer runbook (push → PR → CI → RC script → evidence → tag)

1. **Commit** on a branch (e.g. `release/v1.1.0-rc` or `chore/v1.1.0-prep`) with all intended changes; do not commit local `.coverage` or `services/**/*.db` (see root `.gitignore`).
2. **Push** the branch to `origin` and **open a pull request** to `main` / `master` (GitHub UI or `gh pr create`). *Sole maintainer:* PR is still useful so CI runs on the branch; self-merge after green is fine.
3. Wait until `**[ci.yml](../../../.github/workflows/ci.yml)`** and `**[security-scan.yml](../../../.github/workflows/security-scan.yml)`** complete **successfully** on the PR (or merge commit).
4. On **Linux or macOS**, from repo root: `bash scripts/ci/verify_release_candidate.sh`
  On **Windows**, fix `npm ci` issues (file locks / AV) or run the same Node steps in WSL; the script’s **Ruff** slice targets `services/investigation-agent` only—full `ruff check .` is still the **lint** job in CI.
5. **Fill evidence** in the acceptance checklist tables above (links to Actions runs, PR, screenshots).
6. **Merge** the PR when green. On the **exact merge commit** (or chosen RC SHA), tag:
  `git tag -a v1.1.0 -m "v1.1.0 — see docs/docs/releases/v1.1.0-2026-04-30.md"`  
   `git push origin v1.1.0`  
   Do **not** tag until steps 3–5 are satisfied.

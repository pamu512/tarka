# Release v1.2.0 (Planned)

Planned release date: `2026-05-30`

## Theme

Vertical intelligence starter packs and benchmarkability—**with explicit concentration** on closing competitive gaps called out in [`competitive-critical-review-2026-04.md`](../guides/competitive-critical-review-2026-04.md).

## Strategic concentration (5/30 train)

Engineering and product work for **`v1.2.0` prioritizes these seven pillars** (maps to Day 60 scope in [`roadmap-30-60-90.md`](../guides/roadmap-30-60-90.md) and [`release-gap-closure-schedule.md`](../guides/release-gap-closure-schedule.md)):

| Pillar | Intent for 5/30 |
|--------|------------------|
| **Productized quality** | Opinionated vertical packs, deterministic benchmarks, preset governance profiles—fewer raw knobs, clearer outcomes. |
| **Device / session / integrity** | Stronger packaged tamper/replay/MitM and session signal story across SDKs and inference; consistency controls. |
| **Location & co-presence** | Richer location/co-presence signals and rule-facing keys; documented confidence semantics where not covered by inference proxies alone. **Trunk already has:** optional browser GPS + opt-in IP geo (`ENABLE_IP_GEO_LOOKUP`), `location_context` merge + geo mismatch tags, graph **`Place` / `SEEN_AT`** — v1.2 acceptance should cite tests + operator docs, not re-implement from scratch. |
| **Counters / velocity as a platform** | `feature-service` multi-window counters, normalized velocity keys, **online/offline parity** per [`counter-replay-parity.md`](../guides/counter-replay-parity.md). |
| **Analyst workflows** | Templates, reason-code mappings, tighter decision→review loops; extends v1.1.0 explainability. |
| **Network / consortium effects** | Deepen consortium-weighted intelligence and tenant policy surfaces where they improve collective signal **without** sacrificing tenant isolation. |
| **Operations & lock-in inverse** | Connector reliability scorecards, ingress SLA-style fields, documented deploy profiles and graph-backend alternates—**reduce** operational and legal friction vs closed vendors. |

Deliverables below (starter packs, benchmarks, counters, ingress reliability) **roll up into these pillars**; issue ordering still follows [`oss-ship-order-dependencies.md`](../guides/oss-ship-order-dependencies.md).

### Graph service: Neo4j ↔ JanusGraph (5/30)

**Ship:** `services/graph-service` can run against **Neo4j** (default) or **JanusGraph** via Gremlin Server by setting `GRAPH_BACKEND` and `JANUSGRAPH_*` env vars. The **HTTP API is unchanged**—Decision API, case flows, and investigation / copilot clients keep the same `GRAPH_SERVICE_URL` contract.

- Operator guide: [`services/graph-service/docs/janusgraph-adapter.md`](../../../services/graph-service/docs/janusgraph-adapter.md).
- **Analytics:** Janus mode uses Gremlin + in-process algorithms; fraud-ring detection is documented as an **approximation** vs Neo4j Cypher (same response shape, not guaranteed identical rings).

## DAO integration (scope for 5/30)

**DAO** here means *decentralized community or consortium governance* that should **influence risk signal sharing**, not the in-app “governance” of model promotion (that stays YAML/CI-first in v1.2).

| Layer | Fit with Tarka today | v1.2.0 (5/30) | Later (e.g. v1.3+ / J4) |
|--------|----------------------|---------------|-------------------------|
| **Off-chain DAO → operational policy** | Decisions already support **preset profiles** and **consortium** hooks (`consortium_api`, Redis `fraud:consortium:*`). | **Document** how a DAO’s off-chain votes map to env/config (`CONSORTIUM_*`, trust scores, opt-in participation). Optional **OpenAPI + guide** for `consortium` HTTP APIs. | UI / Trust Center surfacing of consortium posture. |
| **On-chain attestations → consortium signals** | `record_consortium_signal` / `add_consortium_feedback` can carry **provenance** if callers attach signed metadata. | **Spike or thin adapter**: e.g. a small **ingest worker** (or documented `curl` recipe) that turns **signed DAO payloads** (IPFS snapshot, EIP-191 message, or org-specific format) into **consortium API** calls—no chain dependency in core services. | Hardened identity, replay protection, and audit exports (aligns with **v1.3** evidence story). |
| **Token voting inside the product** | No wallet or chain coupling in core stack today. | **Out of scope** for v1.2 unless explicitly prioritized; treat as separate product track. | Revisit if J4 requires on-chain evidence bundles. |

**Dependencies:** stays aligned with the **Network / consortium effects** pillar and [`competitive-critical-review-2026-04.md`](../guides/competitive-critical-review-2026-04.md) “collective signal without sacrificing tenant isolation.” **Tenant isolation** remains non-negotiable: DAO-fed signals are **opt-in** per deployment and must not imply cross-tenant PII leakage.

**Exit criteria for v1.2 DAO slice (pick one or two):**

1. Published **integration guide** under `docs/docs/guides/` describing how DAO outcomes map to consortium APIs (sequence: DAO outcome → consortium endpoints → evaluate behavior) — **[`dao-to-consortium-integration.md`](../guides/dao-to-consortium-integration.md)**.
2. **Reference adapter** under [`scripts/consortium_adapter/`](../../../scripts/consortium_adapter/README.md): CLI + JSON Lines ingest + Python client (`httpx`).
3. If a chain attestation format is chosen, document **one** canonical example (versioned) in `contracts/` or `docs/docs/guides/examples/` — **[`contracts/examples/dao-attestation-v1.example.json`](../../contracts/examples/dao-attestation-v1.example.json)**.

**Reference (factual pattern only, no vendor endorsement):** [`joinsonar-query-feedback-vs-consortium-api.md`](../guides/joinsonar-query-feedback-vs-consortium-api.md) and [`unit21-fraud-dao-vs-consortium-api.md`](../guides/unit21-fraud-dao-vs-consortium-api.md) compare public consortium-style flows to Tarka’s `consortium_api`.

Heavy **J4** (Trust Center, signed release governance) remains **v1.3.0** per [`v1.3.0-2026-06-29.md`](v1.3.0-2026-06-29.md).

### ML explainability (shipping on the 5/30 train)

**Shipped (trunk target for v1.2.0):**

- **`ml-scoring` `POST /v1/score`** returns structured **`ml_top_factors`** (ranked codes + descriptions + impact) and a single-line **`ml_summary`**, derived from existing heuristic / ONNX / adaptive explanations (FraudLens-style *factor list + narrative*, without a runtime dependency on external demos).
- **`decision-api`** passes that slice through **`_fetch_ml_score`** into **`inference_context`**: optional **`ml_top_factors`**, **`ml_summary`**, **`ml_model`** (plus **`driver_reasons`** entries `ml_factor:<CODE>` for the top factors).
- **Contracts:** OpenAPI **`InferenceContext`** in [`contracts/openapi/decision-api.yaml`](../../contracts/openapi/decision-api.yaml); golden example [`contracts/golden/inference-context-v2.example.json`](../../contracts/golden/inference-context-v2.example.json); Python + TypeScript SDK types; Case Detail **ML explanation** block in the frontend.

**Stretch (implemented; optional at runtime, behind extras / env):**

- **TreeSHAP + LightGBM:** `pip install -e "services/ml-scoring[shap]"` (or `tarka-ml-scoring[shap]` from the package), then set **`ML_SHAP_ENABLED=1`** and **`ML_LGBM_MODEL_PATH`** to a **`joblib`**-pickled **`LGBMClassifier` / `LGBMRegressor`** using the same 9-feature order as [`services/ml-scoring/src/ml_scoring/main.py`](../../../services/ml-scoring/src/ml_scoring/main.py). When active, **`POST /v1/score`** uses the tree model score for the adaptive blend base and **`ml_top_factors`** from TreeSHAP. Health JSON includes **`shap_stretch_enabled`**.
- **Batch scoring CLI:** [`scripts/ml_batch_score.py`](../../../scripts/ml_batch_score.py) (CSV → **`POST /v1/score`**); documented in [`scripts/benchmarks/README.md`](../../../scripts/benchmarks/README.md).
- **DAO → consortium:** integration guide [`docs/docs/guides/dao-to-consortium-integration.md`](../guides/dao-to-consortium-integration.md); attestation example [`contracts/examples/dao-attestation-v1.example.json`](../../contracts/examples/dao-attestation-v1.example.json).

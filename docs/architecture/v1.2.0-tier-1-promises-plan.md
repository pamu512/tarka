# v1.2.0: Tier-1 Promises Fulfillment Plan

This plan outlines the phases required to fulfill the strict edge-case handling and resilience promises from the original Tier-1 specification. This branch (`1.2.0`) will serve as the staging ground for these fixes before eventually superseding `master`.

## Phase 1: Resilience & Orchestration Hardening

**1. Event Ingest Poison Pill Handling**

- **Target:** `services/event-ingest/src/main.rs` (`publish_deadletter_record`)
- **Action:** Modify the dead-letter publishing logic to explicitly handle NATS timeouts/failures. If publishing to the deadletter subject fails, log a `CRITICAL` error to stdout. Ensure the original message is still `ack()`ed to prevent infinite crash loops.

**2. Decision API DAG Orchestration**

- **Target:** `services/decision-api/src/decision_api/eval_dag.py`
- **Action:** Replace the current stub `EvalDAG` class with an actual Directed Acyclic Graph implementation.
- **Requirements:**
  - Perform a topological sort on startup.
  - Detect circular dependencies and raise a fatal `ValueError`.
  - Implement short-circuiting: if a parent node fails (e.g., `Feature_Snapshot`), child nodes (e.g., `ML_Score`) must be skipped and marked as `skipped_due_to_dependency_failure`.

## Phase 2: Agentic Hooks & Compliance

**3. Proactive Case Summarization Fallbacks**

- **Target:** `services/case-api/src/case_api/agent_hooks.py` (`fire_case_brief`)
- **Action:** Implement robust error handling for LLM/Agent timeouts.
- **Requirements:**
  - Catch exceptions and retry up to 3 times with exponential backoff.
  - If all retries fail, interact with the Case API database/endpoints to append a Case Comment: `System: Failed to generate automated case brief due to LLM provider unavailability.`

**4. Adverse Action Mapping Strictness**

- **Target:** `services/decision-api/src/decision_api/adverse_action.py`
- **Action:** Enforce FCRA/ECOA compliance rules on adverse action codes.
- **Requirements:**
  - Prioritize codes based on severity rank (e.g., Fraud > Credit > Velocity).
  - Cap the returned codes to a maximum of 4.
  - If a rule triggers without a mapped code, fallback to `G99: Internal Policy` and log a warning.

## Phase 3: Analytics & LLM Reporting

**5. Natural Language to SQL Reporting**

- **Target:** `services/decision-api/src/decision_api/reporting_nl.py`
- **Action:** Implement the promised LLM-based SQL generation.
- **Requirements:**
  - Use an LLM to generate SQL based on the ClickHouse schema instead of hardcoded templates.
  - Implement strict regex/AST parsing to reject queries containing `DROP`, `ALTER`, `INSERT`, or `DELETE`.

**6. Analytics Sink Micro-batching**

- **Target:** `services/analytics-sink/src/main.rs`
- **Action:** Ensure ClickHouse inserts are properly micro-batched. If a batch fails, explicitly `NAK` the NATS messages with a delay to trigger a retry, rather than silently dropping or infinitely looping without backoff.
